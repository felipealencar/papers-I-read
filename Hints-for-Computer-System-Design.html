Hints for Computer System Design

1983 • Design Pattern • Operating Systems • Software Engineering • Engineering • Software • Systems

07 Jan 2022

  • The paper outlines some general hints for system design.

  • Link to the paper

  • Interface design

    • Three conflicting requirements: Interface should be simple, complete, and admit a sufficiently small and fast implementation.

    • Difficulty: Each interface is a small programming language.

    • Hints

      • Do one thing at a time and do it well.

      • Interface should not promise more than the implementor knows to deliver.

      • Make the program fast, rather than making it general or powerful.

      • Abstraction should conceal undesirable properties.

      • Keep basic interfaces stable.

  • Making implementations work

    • Plan to throw away the first implementation.

    • Divide and conquer - reduce a hard problem to many easy ones.

    • Use a good idea again, instead of generalizing it. Sometimes, using a specialized implementation of the idea may be better than using a general one.

  • Handling all the cases

    • Handle normal and worst cases separately as their requirements can differ.

  • Speed

    • Sometimes, it is better to split resources instead of sharing them.

    • Use static analysis.

    • Cache expensive computations.

    • Use “hints” to speed execution. A hint is like a cached value with two differences - the value may be wrong, and it may not be reachable by an associative lookup.

    • When in doubt, use brute force.

    • Whenever possible, compute in the background.

    • Use batch processing.

    • When allocating resources, the first priority should be to avoid overloading the system (and not optimizing it).

  • Fault tolerance

    • Retrofitting fault tolerance to an existing design is very hard.

    • Log updates to record the state of an object.

    • Each log entry should consist of the name of the update procedure and its arguments.

    • The procedure must be a stateless function, and the argument must be valued and not references to mutable objects.

    • Make actions atomic or restartable

      • Atomic actions are either complete or have no effect. If an atomic action fails, there is no need to deal with the intermediate state/effect of the action.

      • A restartable action can be partially executed many times before a complete execution.
